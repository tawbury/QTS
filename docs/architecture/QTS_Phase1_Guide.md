# QTS Phase 1 Guide

**Ops 연동 및 최소 실행 루트 확보 기준 문서**

- File: `QTS/docs/architecture/QTS_Phase1_Guide.md`
    
- Version: v1.0.0
    
- Status: **Enforcing (개발 기준 문서)**
    
- Scope: QTS Phase 1
    
- Last Updated: 2025-12-27
    

---

## 1. 문서의 목적

본 문서는 QTS 프로젝트의 **Phase 1(1차 목표)** 개발을 시작하기에 앞서,  
개발 범위·책임 경계·테스트 위치·폴더 사용 원칙을 **명확히 고정**하기 위해 작성되었다.

본 문서는 Phase 1 기간 동안 다음 역할을 수행한다.

- 개발 방향에 대한 재논의 차단
    
- 구조적 판단의 기준점 제공
    
- 세션 분리 환경에서도 일관된 판단 유지
    
- “지금 이 작업이 Phase 1 범위에 해당하는가”에 대한 판단 근거 제공
    

본 문서는 Phase 1 종료 시점까지 **최상위 기준 문서(Single Source of Truth)** 로 사용된다.

---

## 2. Phase 1 목표 정의

### 2.1 Phase 1의 핵심 목표

Phase 1의 목표는 다음과 같이 정의한다.

> **QTS가 ops 서브 프로젝트의 출력을 입력으로 받아,  
> 실행(Act) 단계까지 포함한 전체 파이프라인을  
> 구조적으로 보유하고 있음을 증명하는 것**

여기서 말하는 “실행”은 **실매매를 의미하지 않는다**.

---

### 2.2 Phase 1에 포함되는 범위

Phase 1에서 반드시 구현·검증해야 할 범위는 다음과 같다.

- ops(Observer / Decision Pipeline) 출력 연동
    
- Decision 결과의 Execution Intent 변환
    
- Broker Engine 최소 구현 (noop 또는 mock)
    
- Runtime 계층을 통한 전체 파이프라인 연결
    
- 테스트를 통한 구조적 증명
    

---

### 2.3 Phase 1에서 제외되는 범위

아래 항목은 Phase 1 범위에서 **의도적으로 제외**한다.

- ops 내부 로직 수정
    
- 실계좌 연동
    
- 실제 주문 전송
    
- 포트폴리오 엔진
    
- 리스크 관리 고도화
    
- 전략 성능 최적화
    
- 오토 스키마 실구현
    
- 구글 스프레드시트 구조 변경
    

---

## 3. Phase 1 개발 흐름

Phase 1은 아래 순서를 따른다.

1. **API 탐색 테스트**
    
    - API의 인증·요청·응답 구조 사실 확인
        
    - QTS 코드에 편입하지 않는다
        
2. **Broker Engine 인터페이스 설계**
    
    - 실행 계층의 책임과 경계 정의
        
    - API 세부사항과 분리된 구조 유지
        
3. **Mock / Noop 기반 Broker 구현**
    
    - 실행 파이프라인 존재 증명 목적
        
    - 실매매는 구조적으로 불가능해야 한다
        
4. **ops 연동 및 Runtime 연결**
    
    - Observer → Decision → Execution → Broker
        
5. **E2E 테스트**
    
    - 전체 파이프라인 1사이클 검증
        

---

## 4. 테스트 기준

### 4.1 테스트 위치 원칙

모든 테스트는 **다음 경로에서만 수행한다.**

```
QTS/tests/
```

- 임시 테스트, 실험 테스트도 예외 없음
    
- 내부 검증을 위해 테스트 파일이 추가되는 경우:
    
    - 검증 종료 후 반드시 이동하거나 삭제한다
        
    - 프로젝트에 잔존시키지 않는다
        

---

### 4.2 테스트 목적 제한

Phase 1의 테스트 목적은 다음으로 제한한다.

- 구조적 연결 검증
    
- 책임 분리 검증
    
- 실행 경로 존재 증명
    

아래 목적의 테스트는 Phase 1에서 수행하지 않는다.

- 성능 측정
    
- 수익 검증
    
- 전략 비교
    

---

## 5. 폴더 사용 기준

### 5.1 `QTS/ops/`

- 종료된 서브 프로젝트로 취급한다
    
- Phase 1 동안 **코드 수정 대상이 아니다**
    
- 연동 테스트를 수행하지 않는다
    
- QTS는 ops를 **외부 의존성(Black Box)** 으로 취급한다
    

---

### 5.2 `QTS/runtime/`

- Phase 1의 주요 개발 대상 영역이다
    
- 실행 계층 및 브로커 엔진 구현은 이 영역에서 수행한다
    

---

### 5.3 `QTS/tests/`

- Phase 1 동안의 **유일한 테스트 실행 위치**
    
- ops 연동 테스트는 반드시 이 경로에서 수행한다
    

---

### 5.4 `QTS/notebook/`

- 개인 작업 공간으로 취급한다
    
- QTS 프로젝트의 일부로 간주하지 않는다
    
- 코딩을 수행하지 않는다
    
- 실험, 메모, API 탐색, 아이디어 정리 용도로만 사용한다
    

notebook의 결과물은 QTS 코드에 직접 편입하지 않는다.

---

### 5.5 `QTS/docs/`

- QTS 관련 모든 문서 저장소
    
- 아키텍처 문서
    
- 기준 문서
    
- 가이드 문서
    
- GitHub 공개 문서
    

docs는 기록과 기준을 위한 공간이며, 실행 코드와 무관하다.

---

### 5.6 `QTS/tools/`

- QTS에 국한되지 않는 범용 도구 제작 공간
    
- 사용 목적 예:
    
    - 코드 일괄 수정
        
    - 폴더 구조 정리
        
    - 로그 및 불필요 파일 정리
        
    - 구조 감사(Audit)
        

tools의 코드는 QTS 런타임 의존성을 가져서는 안 된다.

---

## 6. 구조 변경 원칙

- Phase 1 동안 `ops/` 내부 구조는 변경하지 않는다
    
- Phase 1 동안의 개발은 `runtime/` 과 `tests/` 영역으로 한정한다
    
- 신규 폴더 추가는 다음 조건을 만족해야 한다.
    
    - 책임이 명확할 것
        
    - ops의 책임 영역을 침범하지 않을 것
        

구조 변경이 필요한 경우,  
본 문서를 기준으로 범위 충돌 여부를 먼저 검토한다.

---

## 7. Phase 1 종료 기준 (Gate A)

아래 조건을 모두 만족할 경우 Phase 1 종료를 선언한다.

- ops 출력이 QTS Core로 정상 유입된다
    
- Decision 결과가 Execution Intent로 변환된다
    
- Broker Engine이 이를 수신하고 명시적 응답을 반환한다
    
- 전체 흐름이 테스트로 재현 가능하다
    
- 실매매가 구조적으로 불가능함이 보장된다
    

---

## 8. Phase 2 이후에 대한 원칙

- Phase 2(오토 스키마 구현)는 Phase 1 종료 이후 착수한다
    
- 스키마 구조 변경 및 구글 시트 분리는 완성 단계에서 논의한다
    
- Phase 1 기간 동안 본 문서는 유효한 기준 문서로 유지된다
    

---

## 9. 최종 선언

> **Phase 1 기간 동안 QTS는  
> ‘완성’을 목표로 하지 않는다.
> 
> QTS는 ‘실행 가능한 구조를 가졌음’을  
> 증명하는 데에만 집중한다.**
